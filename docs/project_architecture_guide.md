# プロジェクト設計・アーキテクチャガイド

本リポジトリは、テスト自動化プロジェクトにおける「関心の分離」と「運用効率」を極限まで追求した構成例です。特定の技術（CodeceptJS等）に依存しない、汎用的な設計パターンとして構築されています。

## 1. ディレクトリ構成と役割の定義

各ディレクトリは明確な責務を持ち、相互の依存関係が最小限になるよう設計されています。

- **`tests/` (テストシナリオ層)**:
    - 責務：ビジネスロジックに基づいたテスト手順の記述。
    - 特徴：プロジェクトや機能ごとにサブディレクトリ化され、対象範囲が明確。
- **`pages/` (ページオブジェクト/アクション層)**:
    - 責務：画面操作の具象実装。
    - 特徴：テストシナリオから「UI操作の細部」を隠蔽し、UI変更時の修正箇所を限定する。
- **`env/` (環境定義層)**:
    - 責務：実行環境ごとの変数（URL、アカウント情報等）の管理。
    - 特徴：テンプレート化とプロファイル分離により、機密情報と設定ロジックを切り離す。
- **`scripts/` (開発・運用補助ツール層)**:
    - 責務：開発者のルーチンワークの自動化。
    - 特徴：プロジェクト本体のロジックから独立した、依存関係ゼロのポータブルなツール群（例：`tree_generator.py`）。
- **`docs/` (成果物・ドキュメント層)**:
    - 責務：システムの全体像やテスト結果、設計書の蓄積。
    - 特徴：自動生成された構成図やクラス図が配置され、常に最新の状態が維持される。
- **`support/` (共通基盤層)**:
    - 責務：フレームワークの拡張機能や、全層で利用する共通ユーティリティ。

## 2. 設計上のルールと一貫性

「何をどこに置くか」という問いに対し、本プロジェクトは以下の一貫した基準を持っています。

- **ロジックの排除**: テストファイル（`tests/`）には条件分岐や複雑なループを書かず、読みやすい「手順」のみを記述する。ロジックは `pages/` や `support/` に委譲する。
- **パスの正規化**: すべてのパス参照はルートディレクトリを起点とし、実行環境に依存しないポータビリティを確保する。
- **命名規則の意図**: 実行用バッチは `Run_...`、ドキュメント生成は `docs:...` など、プレフィックスを用いることで、ファイル一覧を見ただけでその役割が判別できる。

## 3. 分離の考え方（デカップリング）

本プロジェクトの最大の強みは、以下の4つの要素が独立している点にあります。

1.  **処理ロジック**: `pages/`, `tests/` に集約。
2.  **設定 (Configuration)**: `env/`, `codecept.conf.js` に集約。
3.  **補助機能 (Tooling)**: `scripts/` に集約。
4.  **成果物 (Artifacts)**: `output/`, `allure-results/`, `docs/generated/` に集約。

この分離により、例えば「テストツールを別のものに変える」際も、補助ツールやドキュメント構造を壊さずに移行することが可能です。

## 4. この構造を採用するメリット

- **オンボーディングの高速化**: `README.md` に常に最新のツリー図とタイムスタンプがあるため、新参者が構造を理解する時間が最短で済む。
- **メンテナンスコストの低減**: 修正が必要な際、「どこを直すべきか」がディレクトリ名から即座に判断できる。
- **開発体験の向上**: `package.json` の整理された `scripts` や、エントリーポイントとなるバッチファイルにより、複雑なコマンドを覚える必要がない。

## 5. 汎用的設計パターンとしての特徴

本プロジェクトの構成は、Web開発、データ分析、インフラ構築など、あらゆるソフトウェアプロジェクトに応用可能な**「ユニバーサル・プロジェクト・テンプレート」**です。

- **ポータビリティ**: `tree_generator.py` のような依存なしツールは、他プロジェクトへそのまま持ち出し可能。
- **スケーラビリティ**: プロジェクトが増大しても、ディレクトリ階層を深くするだけで、全体の秩序を維持したまま拡張できる。
- **可視性の自動化**: 人間に頼らず、コード自身が自分の構造を説明する仕組みを内包している。
