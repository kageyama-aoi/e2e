# 0129_指示文.md 対応案（Allure 二層構造）

## 1. Allure 二層構造の設計意図
- **上位レイヤー（業務視点）**は PM/業務担当向けに「どの業務工程が成功/失敗したか」を最短で判断できる構造にする。
- **下位レイヤー（操作視点）**は テスト担当/開発者向けに、画面遷移・入力値・分岐（breakTarget）の詳細を追える構造にする。
- 両者は排他ではなく共存し、初期表示は俯瞰、クリックで深掘りが自然に進むように Allure の階層（Epic/Feature/Story + step/attachment）で統一する。

---

## 2. 上位レイヤー／下位レイヤーの対応表
- **Epic（業務ドメイン）**: 例）会員管理
- **Feature（業務工程）**: 例）新規会員登録＋経理処理
- **Story（業務フローの分岐）**: 例）正常登録フロー／経理日付エラー（契約日）／経理日付エラー（開始日）
- **step（操作視点）**: 画面遷移・入力・クリック・API待ち・検証など
- **attachment（調査視点）**: 入力値、期待/実際のエラー、スクリーンショット、分岐理由

---

## 3. 修正ポイント一覧
- **Epic/Feature/Story の明示**: テスト起点で Allure ラベルを付与し、業務単位で俯瞰できるように整理する。
- **正常系の step は保持しつつ“控えめ表示”**: step を削らず、Allure の構造で上位層を前面にする。
- **異常系は「テスト成功」扱い**: expectedErrors を尊重し、失敗は `passed` として処理、詳細は attachment へ。
- **breakTarget による分岐の可視化**: 分岐理由や分岐先を attachment に記録して調査を容易にする。
- **既存設計維持**: `buildExecutionPlan / breakTarget / expectedErrors` は変更しない。

---

## 4. 代表的な修正コード例（抜粋）

### Epic / Feature / Story
```js
const allure = codeceptjs.container.plugins('allure');

// 例: テスト開始時
allure.addLabel('epic', '会員管理');
allure.addLabel('feature', '新規会員登録＋経理処理');
allure.addLabel('story', '正常登録フロー');
```

### step / attachment の使い分け
```js
// 操作ログは step で残す（削らない）
I.step('会員基本情報を入力');
I.fillField('氏名', data.name);

// 期待エラーや分岐理由は attachment
allure.addAttachment('期待エラー', expectedError.message, 'text/plain');
allure.addAttachment('入力値', JSON.stringify(data, null, 2), 'application/json');

// 失敗スクリーンショット
const screenshot = await I.saveScreenshot('error.png');
allure.addAttachment('画面キャプチャ', screenshot, 'image/png');
```

---

## 5. なぜこの構造が「俯瞰も深掘りもできる」のか
- 初期表示は Epic/Feature/Story で「業務工程の成否」が即判断できる。
- 詳細確認は step と attachment を展開するだけで「どの入力・画面・分岐で問題が起きたか」へ自然に到達できる。
- step の粒度は維持しているため、安定性・再実行性・既存設計思想を破壊しない。
- 期待エラーを Passed として扱うことで、異常系シナリオも業務工程として完結して見える。
