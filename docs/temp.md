## 宿泊予約および部屋割り当てアクションに伴うデータベース（主に T利用者 テーブル）の更新内容

### 宿泊予約アクションに伴うDB状態の推移（イメージ）

| アクション | 更新対象フィールド (T利用者) | Before (初期状態・未割当) | After (アクション実行後) |
| :--- | :--- | :--- | :--- |
| **一括予約アクション (本集約後)** | 宿泊<br>宿泊室番号<br>宿泊棟手配 | False<br>9999 (通勤者用コード)<br>False | True<br>6501 等の具体的な番号<br>True |
| **個別予約・変更 (前泊・延泊追加)** | 前泊数<br>延泊数 | 0<br>0 | 1 などの追加日数<br>1 などの追加日数 |
| **女性・講師の優先割当 (ロジック適用)** | 性別 または 講師<br>宿泊室番号 | 女 または True<br>未定 | 女 または True<br>7501 (原則7階を割り当て) |
| **満室時の外部宿泊** | 宿泊<br>宿泊室番号<br>備考 | 未定<br>未定<br>空白 | True (宿泊費は発生)<br>9999 (棟内には部屋なし)<br>**「外部ホテル宿泊」**等の記録 |

### 各アクションにおけるDB更新のポイント

1. **一括予約・個別予約のアクション**
    * **Before**: 集約直後のデータは、宿泊の要否が確定していないか、デフォルトで「通勤（宿泊室番号：9999）」として登録されています。
    * **After**: 担当者が「一括予約」を実行すると、システムは空室状況を確認し、**T利用者** テーブルの **宿泊** フラグを **True** に書き換え、具体的な **宿泊室番号** を格納します。これにより、その利用者は「宿泊者リスト」の抽出対象（室番号9999以外）となります。

2. **女性受講者と講師の「7階優先割当」**
    * **DBの判断ロジック**: システムは **T利用者** テーブルの **性別** フィールドが「女」であるか、または **講師** フラグが「True」であるレコードを識別します。
    * **After**: 割当アクションにより、これらの属性を持つ利用者には優先的に **「7XXX」** という7階の部屋番号が **宿泊室番号** フィールドに書き込まれます。7階が満室の場合のみ、他階の番号が書き込まれます。

3. **前泊・延泊の追加アクション**
    * **After**: 受講者管理画面から個別調整を行うと、**T利用者** テーブルの **前泊数** や **延泊数** フィールドに数値が加算されます。
    * **波及効果**: この数値は **Q利用者** クエリを通じて「チェックイン予定日」や「チェックアウト予定日」の計算、さらには請求時の「超過宿泊数」の算出に利用されます。

4. **満室時の「外部宿泊」対応**
    * **Before/After**: 客室数が不足し、受講者が各自で外部ホテルを予約する場合でも、研修費用の計算上は「宿泊あり」として扱う必要があるため、**宿泊** フラグは **True** のまま保持されると考えられます。
    * **After**: ただし、研修センターの宿泊棟には滞在しないため、**宿泊室番号** は **9999** （または外部宿泊を示す特定コード）となり、点呼用の宿泊者リストからは除外される挙動になります。なお、講師については満室時でも原則として宿泊棟内（宿泊室番号あり）に割り当てられます。

### 運用の現状

現在、これらのDB更新アクション（特に一括予約）は、部屋の二重割り当てやデータの不整合を防ぐため、**「Accessを一人しか操作しない」**という運用ルールによって、排他的に処理されています。
