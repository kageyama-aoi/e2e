あなたは E2Eテスト設計に詳しいシニアQAエンジニアです。
以下の CodeceptJS テストコード（syokai_touroku_test.js）を前提に、
「異常系テストが増えても分岐地獄にならない構造」へリファクタしてください。

【現状の課題】
- 経理処理画面Bにおいて、
  正常系（ShouldBeOnKeirisyoriScreenB）と
  異常系（ShouldBeOnKeirisyoriScreenBWithValidationErrors）が
  ほぼ同じ処理内容で関数分離されている
- 今後、開始日・契約日・クラス選択・コース設定など、
  経理処理B内の異常系を1項目ずつ網羅的に増やしたい
- このまま異常系ごとに関数や if/else を増やすと、
  テストコードが仕様を再実装してしまい、保守不能になる

【ゴール】
- 経理処理画面Bの操作は「1本の共通関数」に統合する
- 正常系／異常系の違いは「データ定義（CSV）」で表現する
- テストコード側の分岐は「壊すポイントの注入」に限定する
- 異常系が10個、20個増えても、テストロジックが増えない構造にする

【設計ルール】
1. 経理処理Bの基本フローは常に同じにする
   - クラス選択
   - クラス適用
   - 契約日・開始日入力
   - コース料金設定
2. 異常系は「どこを壊すか（breakTarget）」と
   「どう壊すか（value）」をデータで指定する
3. breakTarget に応じて
   - 入力値を上書きする
   - その操作だけスキップする
   という最小限の分岐のみ許容する
4. if/else をエラーケースの数だけ増やしてはいけない

【期待するアウトプット】
- ShouldBeOnKeirisyoriScreenB / WithValidationErrors を統合した新関数
- 日付入力を「エラー注入対応」にした関数
- エラー検証を共通化した関数
- Scenario 側が分岐なしで読める最終形
- なぜこの構造にすると異常系に強くなるのかの説明

【重要】
- 現在の画面遷移・UI操作の流れは変えない
- CodeceptJS の流儀を尊重する
- テストは「業務フローの破壊」を検証するものであり、
  バリデーションロジックをテスト側に書かないこと
